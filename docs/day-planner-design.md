# Design: Day Planner + Class Schedule

Status: **design spec** (not yet implemented). Targets the existing Focl
codebase — React + Vite + Capacitor, pure reducer in `src/state/store.js`,
tokens in `src/lib/theme.js`, agenda helpers in `src/lib/helpers.js`.

Decisions locked with the user:
- **Block model**: timeboxed timeline (start/end clock times, drag + resize).
- **Lock/emergency**: the next day locks once it begins; emergency changes are
  **unlimited but logged and counted in stats**.
- **Class Schedule**: a **dedicated data type** with a weekly-grid editor in
  Settings (mirrors how Spaces lives under Settings).

---

## 1. Concept

At night you **build tomorrow** on a timeline: drag tasks, classes, and
reusable templates into timed blocks. The builder is **only openable 8PM–12AM**
(the "planning window"). Once tomorrow begins, the plan **locks** — you execute
it, you don't redesign it. You *can* force a change during the day, but every
forced change is logged and shows up in your stats as a plan-discipline signal.
Adherence (planned vs. actually done, and on-time) feeds the weekly archive and
Insights.

**Class Schedule** is a recurring weekly timetable. Classes auto-populate the
planner timeline (and the Plan agenda) on their weekdays as fixed anchors you
build around, and can fire start reminders.

### Naming note (avoid collisions)
The codebase already has two "plan"s: the workout weekly split (`plan` slice)
and the **Plan** agenda nav screen. To avoid confusion:
- New slice is **`dayPlans`** (per-date timelines).
- The feature is surfaced as **"Tomorrow"** inside the existing **Plan** screen,
  not a new nav tab (the bottom nav is full: Today · Plan · + · Habits · Train).

---

## 2. Where it lives in the app

- **Plan screen gains a segmented control: `Agenda` | `Tomorrow`.**
  `Tomorrow` is the Day Planner builder. The week-strip still works; selecting a
  future day shows its (read-only unless in-window) planned timeline.
- **Today screen** gets a **night CTA card** that only renders during the
  planning window: *"Planning time — build tomorrow ›"* → deep-links to
  Plan ▸ Tomorrow. Outside the window the card is absent.
- **Settings** gains a **Class Schedule** row (like Spaces) → weekly-grid editor.
- **Insights** gains a **Plan adherence** section.

---

## 3. Data model (reducer slices + actions)

All new slices are persisted via `PERSISTED_KEYS` and pure-reducer actions, same
pattern as everything else. Ids generated by callers (`newId()`); side effects
(notifications/haptics) stay in `App.jsx`.

### 3.1 `dayPlans` — one record per planned date
```js
{
  date: '2026-06-26',            // the day being planned (localISO)
  status: 'draft' | 'locked',   // draft while in planning window; locked once the day starts
  committedAt: '2026-06-25T21:40:00.000Z' | null,
  blocks: [
    {
      id, title,
      start: '07:30', end: '08:15',   // 24h, snapped to 15-min grid
      kind: 'task' | 'class' | 'custom',
      refId: null,                    // taskId for kind:'task', classId for kind:'class'
      spaceId, color,                 // color defaults from space/class
      done: false,
      sourceTemplateId: null,         // blockTemplate it came from, if any
      reminder: { lead: '10min' } | null,
    }
  ],
  emergencyEdits: [                   // audit log — the intentionality cost
    { at: ISO, action: 'add'|'move'|'resize'|'remove'|'edit', blockId, note }
  ],
}
```

### 3.2 `blockTemplates` — reusable single blocks
A task added "as template" becomes one of these; "one-time" tasks are plain
blocks with no `sourceTemplateId`.
```js
{ id, title, durationMin: 45, kind: 'task'|'custom', spaceId, color,
  defaultStart: '07:00' | null, reminder: { lead } | null }
```

### 3.3 `dayTemplates` — named full-day layouts
```js
{ id, name: 'Weekday', blocks: [ { title, start, end, kind, refId, spaceId, color } ] }
```
Applying a day template to a date seeds that date's `dayPlans.blocks`
(classes for that weekday are merged in automatically — see §6).

### 3.4 `classes` — dedicated weekly timetable
```js
{ id, name: 'Algorithms', spaceId, color,
  days: ['Mon','Wed','Fri'],      // weekday keys (existing convention)
  start: '11:00', end: '12:30',
  location: 'Hall B' | '',
  teacher: '' ,
  weeks: 'all' | 'odd' | 'even',  // for biweekly timetables; default 'all'
  reminder: { lead: '30min' } | null,
  active: true,                   // soft-disable without deleting
}
```

### 3.5 `settings` additions
```js
planningWindow: { start: '20:00', end: '24:00' },  // configurable; default 8PM–12AM
dayPlannerEnabled: true,
```

### 3.6 Actions
```
dayplan/seed            { date, blocks }          // from day template + classes
dayplan/addBlock        { date, block }
dayplan/updateBlock     { date, id, patch }       // move/resize/edit
dayplan/removeBlock     { date, id }
dayplan/commit          { date }                  // status -> 'locked', stamps committedAt
dayplan/emergencyEdit   { date, entry, patch... } // wraps an edit + appends to emergencyEdits
dayplan/toggleBlockDone { date, id }

blockTemplate/save  { template }     blockTemplate/delete { id }
dayTemplate/save    { template }     dayTemplate/delete   { id }    dayTemplate/apply { id, date }

class/save  { class }                class/delete { id }
```
`reducer` stays pure; `dayplan/emergencyEdit` is the *only* mutator allowed once
`status === 'locked'`, and it always appends to `emergencyEdits`.

## 4. The planning window + lock (pure, testable)

New helpers in `helpers.js`, fully unit-testable like the rest of `state/`:

```js
// Is `now` inside the planning window? Window may cross/end at midnight.
isPlanningWindow(now, window)      -> bool

// The date the planner is currently building (tomorrow, during the window).
plannableDate(now, window)         -> 'YYYY-MM-DD' | null

// Given a dayPlan + now, what may the user do?
planEditMode(dayPlan, now, window) -> 'edit' | 'locked' | 'readonly'
//   edit     = in window for that date (free editing)
//   locked   = the day is current/past, plan committed -> emergency-only
//   readonly = a future date not yet in its window
```

Lifecycle for a date D:
1. **D-1, 20:00–24:00** → the window for D. `status:'draft'`, free editing.
2. **D-1, 24:00 (or first open on D)** → auto-`commit`: `status:'locked'`. This is
   done by `runDailyMaintenance` (the existing once-per-day startup pass), so no
   timers needed — it locks any draft whose date is now today/past.
3. **During D** → `locked`. Edits require the **Emergency change** affordance,
   which routes through `dayplan/emergencyEdit` and is logged.
4. **After D** → archived into stats (§7).

Gating UI: outside the window, Plan ▸ Tomorrow shows a calm empty state —
*"The planner opens at 8:00 PM. Rest up."* — with a live countdown. The builder
controls are simply not mounted, so there's no way to edit.

## 5. Day Planner builder (timeline DnD)

A vertical timeline (configurable visible range, default **06:00–24:00**), one
block per timed item. No new DnD library — reuse the app's hand-rolled pointer
gestures (the same approach the old SwipeRow used), with **15-minute snapping**.

- **Timeline**: hour gridlines; a "now" line on the current day.
- **Block card**: `s1` surface, `border-left: 3px {color}`, title + time range;
  classes render with a small "CLASS" tag and are **pinned** (can't be moved off
  their class time, but you can build around them).
- **Drag** the body to move; **drag the bottom edge** to resize; both snap to
  :00/:15/:30/:45. Overlaps are allowed but flagged with a hairline warning
  stripe (you may double-book; the app just tells you).
- **Palette drawer** (bottom sheet): three source tabs —
  - **Tasks**: existing undated tasks + a quick-create field. Drag onto the
    timeline to schedule (creates a `kind:'task'` block linked by `refId`).
    Adding offers **"Save as template"** (→ `blockTemplates`).
  - **Templates**: your `blockTemplates`, drag to drop at a time.
  - **Classes**: today-relevant classes (auto-added, listed for reference).
- **Day templates**: a top action — **Apply template** (pick a `dayTemplate` →
  seeds the timeline, merging that weekday's classes) and **Save day as
  template** (snapshots the current blocks to a named `dayTemplate`).
- **Commit**: a primary **"Lock in tomorrow"** button. Until pressed, the plan is
  a draft; pressing it stamps `committedAt` (it still auto-locks at midnight if
  you forget).

Interaction tokens reuse the redesign: 14–18px radii, `accent` primary,
`s1/s2/s3` surfaces, DM Mono for times.

## 6. Class Schedule

- **Editor in Settings** → a **weekly grid** (Mon–Sun columns, time rows) plus a
  list editor. Add/edit a class: name, space (color), days, start/end, location,
  optional teacher, `weeks` (all/odd/even), reminder lead.
- **Feeds**:
  - **Day Planner**: when seeding/opening a date's plan, classes whose `days`
    include that weekday (and whose odd/even matches) are injected as pinned
    `kind:'class'` blocks.
  - **Plan agenda**: `agendaForDay` is extended to merge classes (a new
    `classOccursOn(class, dateISO)` helper, sibling to `eventOccursOn`).
  - **Today "Up next"**: classes participate alongside events.
- **Reminders**: a class with a reminder schedules a recurring custom-day
  notification via the existing `buildNotificationSpecs` path (treat a class like
  a recurring event with `customDays = class.days`). No new notification engine.

## 7. Stats integration

Planning is only meaningful if it scores. Extend the weekly archive snapshot
(`buildWeekSnapshot` in `state/maintenance.js`) and Insights:

Per day (derived, not stored twice):
- `blocksPlanned`, `blocksDone`, `adherence = done/planned`.
- `onTimeDone` — block checked done within `[start, end+grace]` (timeboxing makes
  this possible; grace default 30min).
- `emergencyEdits` count.

Weekly archive gains: `planAdherencePct`, `blocksPlanned`, `blocksDone`,
`onTimePct`, `emergencyEdits`. Insights gains a **"Plan adherence"** card
(adherence ring + on-time % + "N emergency changes this week", framed as a
discipline metric, not a scold).

Tasks completed via a planner block still flow through the existing
`task/toggle` (a `kind:'task'` block's checkbox toggles its linked task), so
task stats stay consistent — no double counting.

---

## 8. Notifications (sticky-until-finished)

Extends the rule already shipped for tasks ("ongoing, cleared on completion"):
- **Planner block reminders** and **class reminders** schedule as `ongoing:true`
  (non-swipeable). Marking the block done (or the class' linked task) cancels it.
- Reuses `buildNotificationSpecs`; add `kind:'class'`/block handling so the
  `ongoing`/`autoCancel` flags and the done-aware skip apply uniformly. The
  existing `MAX_NOTIFS_PER_ITEM` cap and startup `resyncAll` cover them too.

---

## 9. Edge cases & rules

- **Timezone**: every date is `localISO`; the window math uses local time.
- **Never planned**: if no `dayPlans` record exists for today, Today behaves
  exactly as it does now (ad-hoc tasks). The planner is additive, never blocking.
- **Window miss**: if you skip planning, tomorrow simply has no plan (or only its
  classes). No penalty beyond an empty adherence stat.
- **Emergency during the day**: any locked-plan edit opens a small confirm
  ("This counts as an emergency change") → logs to `emergencyEdits`. Unlimited.
- **Editing a class** affects future occurrences; a one-off "skip this class on
  date D" is a per-dayPlan block removal (logged if locked), not a class edit.
- **DST / midnight window**: `isPlanningWindow` handles `end:'24:00'` as
  end-of-day; tested explicitly.
- **Overlaps**: allowed, visually flagged, never blocked.

## 10. Phased build plan

1. **Data & logic (no UI)** — `dayPlans`/`blockTemplates`/`dayTemplates`/`classes`
   slices + actions; `isPlanningWindow`/`plannableDate`/`planEditMode`/
   `classOccursOn`/adherence helpers; auto-lock in `runDailyMaintenance`. Full
   unit tests. *Shippable, invisible.*
2. **Class Schedule** — Settings weekly-grid editor; merge classes into
   `agendaForDay` + Today "Up next"; class reminders. *Usable on its own.*
3. **Day Planner builder** — Plan ▸ Tomorrow timeline, DnD + snapping, palette
   drawer, block/day templates, window gating + Today night CTA.
4. **Lock + emergency log + stats** — commit/auto-lock, emergency flow, archive
   fields, Insights "Plan adherence" card.
5. **Notifications** — sticky block/class reminders through the existing engine.

Each phase is independently shippable behind the existing lint/test/build CI,
and none of it touches the persisted shape of existing slices — it only adds new
ones, so current data migrates untouched.

---

## 11. Open defaults (chosen, easy to change)

- Visible timeline range **06:00–24:00**; snap **15 min**; on-time grace **30 min**.
- Planning window **20:00–24:00**, configurable in Settings.
- Classes are pinned in the planner (build-around), not freely movable.
- Day Planner is a **mode inside Plan**, not a new nav tab.
